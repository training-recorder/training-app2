## 1. PWA（Progressive Web App）とは

### 概要
通常のウェブサイトに「アプリっぽい機能」を追加したもの。
- スマホのホーム画面にインストールできる
- オフラインでも動く
- ブラウザのアドレスバーが消えてアプリっぽく見える

### 必要なもの
| ファイル | 役割 |
|---|---|
| `manifest.json` | アプリ名・アイコン・テーマ色などの定義 |
| `sw.js`（Service Worker） | オフライン対応・キャッシュ管理 |

### Service Worker の仕組み
```
通常のアクセス：ブラウザ → サーバー → ファイル取得

Service Worker あり：
ブラウザ → Service Worker（仲介役）→ キャッシュ or サーバー
```
- 初回アクセス時にファイルをキャッシュ（保存）
- 次回からはキャッシュから返す → オフラインでも動く
- キャッシュ名（v1, v2...）を変えると古いキャッシュが更新される

## 2. IndexedDB とは

### 概要
ブラウザに内蔵されているデータベース。
- サーバー不要でデータをローカルに保存できる
- リロードしてもデータが残る
- JSON形式でデータを保存できる

### 今回のストア構成
```
DB名: trainingApp（version 1）
├── plans      … 取り込んだメニュー（予定）
├── records    … 実際の記録（実績）
├── videoStock … ジャンル別動画URL
└── settings   … 長期目標などの設定
```

### 重要なポイント
- DBのバージョンを最初から決めておく（後から変えると移行が大変）
- 今回は v1 で全ストアを作り、バージョン変更なしで運用できるようにした

## 3. GitHub と GitHub Pages

### GitHub とは
コードを保存・管理するためのクラウドサービス。
- **リポジトリ**：コードをまとめたフォルダのようなもの
- **git**：バージョン管理ツール（変更履歴を記録する）

### GitHub Pages とは
GitHubのリポジトリを**無料でウェブサイトとして公開**できる機能。
- `https://ユーザー名.github.io/リポジトリ名/` という URL でアクセスできる
- 今回は `https://training-recorder.github.io/training-app2/`

### 基本的な git コマンド
```bash
git init                    # リポジトリを初期化
git add .                   # 変更ファイルをステージング
git commit -m "メッセージ"   # 変更を記録
git push origin main        # GitHub に送信
git remote -v               # リモートURLを確認
git status                  # 現在の状態を確認
```

## 4. 今回うまくいかなかったところ（GitHub 編）

### ❌ トラブル1：Jekyll ビルドエラー

**何が起きたか：**
GitHub Pages のビルドが失敗し、メールで通知が届いた。

**原因：**
GitHub Pages はデフォルトで **Jekyll**（ブログ用のビルドツール）を使ってビルドしようとする。
しかし今回のアプリはバニラ HTML / CSS / JS なので、Jekyll は不要で、むしろ邪魔だった。

**解決方法：**
リポジトリのルートに `.nojekyll` という**空のファイル**を置く。
これで「Jekyll を使わないでください」という指示になる。

```bash
echo. > .nojekyll
git add .nojekyll
git commit -m "Add .nojekyll"
git push origin main
```

**教訓：**
GitHub Pages でバニラ HTML を公開するときは、必ず `.nojekyll` を置く。

---

### ❌ トラブル2：リポジトリ構造の問題

**何が起きたか：**
デプロイはできたが、アプリが 404 エラーで表示されなかった。

**原因：**
リポジトリの構造が以下のようになっていた：
```
training-app/（リポジトリルート）
  └── トレーニングアプリ/   ← サブフォルダ
        ├── index.html      ← ここにある
        ├── css/
        └── js/
```

GitHub Pages は `index.html` が**リポジトリのルート**にある必要がある。
サブフォルダの中にあっても認識されない。

**解決方法：**
新しいリポジトリ（`training-app2`）を作り、`トレーニングアプリ/` フォルダの**中**で `git init` して push した。
```
training-app2/（リポジトリルート）
  ├── index.html   ← ルートに直接ある
  ├── css/
  └── js/
```

**教訓：**
GitHub Pages を使うときは、`index.html` がリポジトリのルートに直接あることを確認する。

### ❌ トラブル3：アカウント名変更による URL の不一致

**何が起きたか：**
`git push` で「Repository not found」エラーが出た。

**原因：**
GitHub のアカウント名を変更したため、`git remote` に登録されている URL が古いままになっていた。

- 変更前：`https://github.com/pescados0de0abvssal-wq/training-app.git`
- 変更後：`https://github.com/training-recorder/training-app.git`

**解決方法：**
`git remote set-url` で URL を更新する。
```bash
git remote set-url origin https://github.com/training-recorder/training-app2.git
```

確認：
```bash
git remote -v
# origin  https://github.com/training-recorder/training-app2.git (fetch)
# origin  https://github.com/training-recorder/training-app2.git (push)
```

**教訓：**
GitHub のアカウント名・リポジトリ名を変更したら、ローカルの `git remote` も更新する。

### ❌ トラブル4：CMD でのコマンドの改行問題

**何が起きたか：**
長い URL を含むコマンドを実行しようとしたら、途中で改行されて分断された。

例：
```
git remote add origin https://github.com/training-
recorder/training-app2.git
```
↑ これは2行として認識される。`recorder` は別のコマンドとして実行されてエラーになる。

**解決方法：**
- チャットからコマンドを**1行ずつ丸ごとコピー**する
- CMD に**右クリック → 貼り付け**
- Enter を押す前に、**1行で表示されているか確認**する

**教訓：**
CMD では改行 = コマンドの区切り。長いコマンドは特に注意。

## 5. ローカルサーバーが必要な理由

### なぜ `file://` で直接開いてはダメなのか

Service Worker と IndexedDB は **Secure Context**（安全な通信環境）でしか動かない。
- ✅ `http://localhost:...`（ローカルサーバー）
- ✅ `https://...`（HTTPS）
- ❌ `file:///C:/...`（ファイルを直接開く）

### ローカルサーバーの起動方法
```bash
# npx（npm がインストール済みの場合）
npx serve

# Python
python -m http.server 8000
```

## 6. 今回のアーキテクチャの工夫

### API キー不要の設計
通常 AI を使うアプリは API キーが必要で、サーバーも必要になる。
今回は「Claude チャットに貼り付けるプロンプトを出力する」方式にしたことで：
- API キー不要
- サーバー不要
- 課金管理不要
- GitHub Pages（無料）だけで完結

### 予定と実績を分けて持つ
- `plans`（予定）と `records`（実績）を別のストアで管理
- メニュー通りにいかなくても記録が壊れない
- やれなかった日も正直に記録できる
