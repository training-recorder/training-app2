## ステージ5：セキュリティチェック & 修正（Claude Code）

**実施日：** 2026年6月16日

**手法：** 全コード手動精査 + XSS・注入攻撃を中心とした脆弱性診断 + エンドツーエンド検証

### 検出脆弱性：3点

| # | 深刻度 | 箇所 | 内容 |
|---|--------|------|------|
| 1 | 高 | `home.js:46` | `totalMinutes` がエスケープ・検証なしで `innerHTML` に挿入 |
| 2 | 中 | `home.js:175` / `videos.js:71` | URLを `href` に出力する際、`escHtml` のみで `javascript:` スキーム対策なし |
| 3 | 中 | `index.html` | CSP（Content-Security-Policy）未設定 |

### 脆弱性の詳細

**脆弱性1: totalMinutes XSS（高深刻度）**

Claudeチャット出力のJSON（外部由来）をユーザーが貼り付ける運用のため、`validatePlan()` が `totalMinutes` を検証していないと細工されたJSONで任意のHTMLを注入可能。

実証例：
```json
{
  "plan": {
    "title": "x",
    "weekNumber": 1,
    "days": [
      {
        "day": 1,
        "menu": [],
        "totalMinutes": "<img src=x onerror=alert(document.cookie)>"
      }
    ]
  }
}
```
上記を取り込むと `home.js:46` で生のHTMLとして挿入され、スクリプトが実行される。

**脆弱性2: javascript: スキーム対策なし（中深刻度）**

動画URLを `href` に出力する際、`escHtml()` だけでは `javascript:` スキームを無害化しない。`href="javascript:alert('xss')"` が実行される可能性。

**脆弱性3: CSP未設定（中深刻度）**

XSSが混入した場合の多層防御がない。

### 修正内容

**修正1: util.js に safeUrl() 関数を追加**
```javascript
function safeUrl(url) {
  try {
    const parsed = new URL(url);
    return (parsed.protocol === 'http:' || parsed.protocol === 'https:') ? url : '#';
  } catch (e) {
    return '#';
  }
}
```
- `http://` / `https://` のみ許可
- その他のスキームは `#` に変換

**修正2: home.js で escHtml / safeUrl を適用**
- `home.js:46` — `totalMinutes` に `escHtml()` を適用
- `home.js:175` — href に `safeUrl()` を適用

**修正3: videos.js:71 で safeUrl を適用**

**修正4: index.html に CSP メタタグを追加**
- インラインスクリプト・スタイル全面禁止
- 外部CDNからのロード禁止
- 自オリジン（`'self'`）のリソースのみ許可

**修正5: sw.js キャッシュバージョンを v4 → v5 に更新**

### 検証結果

- 悪意あるJSONをDB投入 → ホーム描画 → スクリプト未実行（`window.__pwned: 0`）を確認
- `totalMinutes` / `title` がテキストとして表示（HTMLとして解釈されず）
- `javascript:` URLが `href="#"` に無害化
- CSPがインラインスクリプトをブロック（第2防御層も機能）
- 既存機能への影響なし（記録・履歴・動画・リンク全て正常）

### 運用上の原則

今後の修正・機能追加で脆弱性が再発しないよう、以下を原則とする：

1. `innerHTML` に動的値を入れない → 必ず `escHtml()` を適用
2. `href` に動的URL → 必ず `safeUrl()` を適用
3. JSON取り込み時 → `validatePlan()` で型・値の範囲を厳格に検証
4. ユーザー入力 → 表示前に必ずエスケープ

**確認済み項目：**
- カレンダーで過去日付タップ→詳細表示に編集ボタン表示
- 編集ボタン押下でホーム同等の入力フォーム（チェックボックス・メモ）に切り替わる
- メモ入力→保存→詳細表示に正しく反映、カレンダーのドット（has-record）も更新
- 未来日付では「未来日付は編集できません」と表示され編集不可
- メニューがない日でもメモのみ編集可能

### トラブルと解決

**問題：preview評価でメモが空のまま保存される**
- 原因：DOM上にホーム画面用と編集フォーム用、同じ `.rf-note` クラスの要素が2つ存在しており、テストスクリプトの `preview_fill` がホーム側の要素を誤って選択していた
- 影響範囲：テストコードのみ。`saveRecordForDate()` は `containerEl` スコープで要素を取得しているため、実際の編集フォーム操作では問題なし
- 教訓：今後同様の共通フォームを複数箇所に表示する場合、クラス名だけでなくコンテナスコープでの要素取得を徹底する（実装側は既に対応済み）

### 運用上の原則（追加）
- ホーム画面・履歴編集画面など、同じフォームを複数箇所に描画する場合は、要素取得を必ずコンテナ（`containerEl`）スコープで行う
